# OpenCode Memory System — Design Document

## Philosophy
Memory should feel natural, not bolted on. Two tiers:

1. **Session Memory** — each session remembers what happened in it (already exists via compaction, but we make it richer)
2. **Global Memory** — a persistent, cross-session knowledge base that grows over time

## Architecture

### Tier 1: Session Memory (Per-Session)
OpenCode already has compaction (summary when context overflows). We enhance this:

- **After every completed task** (not just on overflow), generate a lightweight "memory snapshot"
- Stored as structured JSON in the existing Storage system: `["memory", "session", sessionID]`
- Contains:
  ```json
  {
    "sessionID": "ses_xxx",
    "title": "Fixed auth bug in login flow",
    "created": 1710000000000,
    "updated": 1710003600000,
    "summary": "Fixed JWT validation bug where expired tokens were accepted...",
    "decisions": ["Switched from HS256 to RS256", "Added token refresh endpoint"],
    "files_touched": ["src/auth/jwt.ts", "src/routes/login.ts"],
    "learnings": ["The jose library requires explicit algorithm whitelist"],
    "tags": ["auth", "security", "jwt"]
  }
  ```
- Generated by the existing compaction/summary agent (reuse `summary` agent) at session end or on `/compact`
- This replaces nothing — it's additive. Existing compaction still works for context overflow.

### Tier 2: Global Memory (Cross-Session)
A persistent memory store that the agent can read at session start and write to during work.

**Storage:** File-based in `~/.local/share/opencode/memory/`
- `global.json` — the curated memory (like MEMORY.md but structured)
- `index.json` — searchable index of all session memories

**Structure of `global.json`:**
```json
{
  "version": 1,
  "updated": 1710003600000,
  "entries": [
    {
      "id": "mem_xxx",
      "created": 1710000000000,
      "category": "preference",
      "content": "User prefers RS256 over HS256 for JWT signing",
      "source": "ses_xxx",
      "relevance": 0.9
    },
    {
      "id": "mem_yyy",
      "created": 1710000000000,
      "category": "project_knowledge",
      "content": "The auth system uses a custom middleware chain: rate-limit → cors → jwt → rbac",
      "source": "ses_yyy",
      "relevance": 0.8
    }
  ]
}
```

**Categories:**
- `preference` — user likes/dislikes, coding style preferences
- `project_knowledge` — architectural decisions, how things work
- `lesson` — mistakes made, things learned
- `context` — ongoing projects, goals, background info
- `decision` — why something was done a certain way

### How It Works

#### On Session Start
1. Load `global.json` (or top N entries by relevance/recency)
2. Inject as system prompt section: `<memory>...</memory>`
3. Also load last 3 session summaries from `index.json` for recent context

#### During Session
- New tool: `MemoryWrite` — agent can explicitly save something to global memory
  - Parameters: `{ content: string, category: string }`
  - Called when agent learns something worth remembering
- New tool: `MemorySearch` — agent can search past session memories
  - Parameters: `{ query: string, limit?: number }`
  - Searches session memory index by keyword/fuzzy match

#### On Session End (or after significant task completion)
- Background process runs the `summary` agent on the session
- Extracts structured session memory (Tier 1)
- Optionally promotes important findings to global memory (Tier 2)
- Uses a lightweight LLM call to decide what's worth keeping globally

### Memory Consolidation (Background)
Periodically (or on explicit command), a consolidation pass runs:
- Merges duplicate/similar entries in global memory
- Decays old entries (reduce relevance score)
- Removes stale entries (relevance < 0.1 or older than configurable threshold)
- Keeps global memory bounded (default: 200 entries max)

## Telegram Integration — Design

### Architecture
Telegram is NOT a plugin. It's a **client** — like the TUI or the web UI. It connects to OpenCode's HTTP server (same as `opencode run --attach` does) via the SDK.

### How It Works

```
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│ Telegram │────▶│  Telegram    │────▶│  OpenCode    │
│  User    │◀────│  Bridge      │◀────│  Server      │
└──────────┘     └──────────────┘     └──────────────┘
                  (new command)        (existing serve)
```

**`opencode telegram`** — a new CLI command that:
1. Starts OpenCode server (or attaches to existing one via `--attach`)
2. Connects to Telegram Bot API via long polling
3. Maps Telegram chat → OpenCode session
4. Forwards messages both ways

### Session Mapping
- Each Telegram user has ONE active session at a time — **persisted across restarts**
- Session state stored in `~/.local/share/opencode/telegram/sessions.json`:
  ```json
  {
    "chat_123456": {
      "sessionID": "ses_xxx",
      "created": 1710000000000,
      "updated": 1710003600000
    }
  }
  ```
- **First ever message** → auto-creates a new session, maps it
- **Come back anytime (next day, next week)** → still in that same session, just continue chatting
- **`/new`** → only way to create a new session. Old session preserved in OpenCode, Telegram routes to new one
- **`/session`** → lists recent sessions, lets user switch to any
- **`/status [n]`** → peek at any session's progress without switching

### Message Flow

**Incoming (Telegram → OpenCode):**
1. User sends message in Telegram
2. Bridge looks up mapped sessionID for this chat
3. If no session exists, creates one via SDK
4. Calls `sdk.session.prompt({ sessionID, parts: [{ type: "text", text: message }] })`
5. Subscribes to SSE events for this session

**Outgoing (OpenCode → Telegram):**
1. Bridge listens to SSE event stream (same as `run` command does)
2. On `message.part.updated` with `type: "text"` and `time.end`:
   - Sends text to Telegram chat
3. On tool use completion:
   - Sends brief inline notification (e.g., "📁 Read src/auth.ts")
4. On error:
   - Sends error message to Telegram

### TUI + Telegram Are Fully Independent
- **Telegram** and **TUI** are separate clients. Each tracks its own "active session" independently.
- **TUI** always opens a fresh new session on launch (default OpenCode behavior, unchanged).
- **Telegram** persists its active session across restarts — you come back next day, still in the same session.
- **Sessions are just sessions.** No special "telegram" type. A session created from Telegram shows up in `/session` list on TUI and vice versa.
- **No shared active-session state.** Telegram being in session A doesn't affect what TUI shows.
- **Both CAN work on the same session** if user explicitly switches to the same one — OpenCode server handles concurrent clients fine via SSE events.

### Telegram Commands
- `/new` — create new session (only way to start fresh)
- `/session` — list recent sessions, tap to switch
- `/session [id]` — switch to specific session
- `/status` — show all active sessions and their state
- `/status [n]` — peek at a specific session's progress
- `/cancel` — cancel current operation
- `/model [provider/model]` — switch model

### Output Formatting
- **Normal messages** → full text response, same as TUI
- **Tool use** → condensed one-liners (e.g., "📁 Read src/auth.ts", "✏️ Edit jwt.ts")
- **Long outputs** → truncated to Telegram's 4096 char limit, split into multiple messages if needed
- **Code blocks** → preserved as monospace
- **No streaming** — wait for complete response, then send (cleaner on mobile)

### File Handling
- Photos/documents sent via Telegram → attached as file parts
- Code blocks in responses → sent as monospace text
- Long outputs → truncated with "..." and full output available via `/output`

### Implementation

**New files:**
- `src/telegram/index.ts` — Telegram bot client (grammy or raw API)
- `src/telegram/bridge.ts` — Message routing, session mapping, SSE subscription
- `src/telegram/format.ts` — Markdown → Telegram markdown conversion
- `src/telegram/commands.ts` — Telegram command handlers
- `src/cli/cmd/telegram.ts` — CLI command `opencode telegram`

**Dependencies:**
- `grammy` — Telegram Bot framework (lightweight, TypeScript-native)
  OR raw `fetch` against Bot API (zero deps, more control)

**Recommendation:** Use raw `fetch` + long polling. Keeps it zero-dependency and consistent with OpenCode's style. Grammy is nice but adds weight.

### Why Not a Plugin?
Plugins in OpenCode are hooks that modify behavior (auth, tool definitions, system prompts). Telegram is a full client with its own lifecycle, UI formatting, session management. It doesn't fit the plugin model — it fits the client model (like TUI, web, desktop).

## Key Design Decisions

1. **File-based, not DB** — Memory uses JSON files via existing Storage namespace, not SQLite. Simpler, portable, inspectable.

2. **Agent-driven extraction** — We don't try to heuristically extract memories. The LLM decides what's worth remembering, using a dedicated prompt.

3. **Bounded and decaying** — Memory doesn't grow forever. Old/irrelevant entries fade. This keeps injection cost low.

4. **Non-invasive** — Everything hooks into existing systems (Storage, Plugin hooks, Agent prompts, Tool registry). No schema migrations needed.

5. **Opt-in but default-on** — Config flag to disable. On by default because it's the whole point.

6. **Session memory is structured** — Not free-text blobs. Structured fields enable search and filtering.
